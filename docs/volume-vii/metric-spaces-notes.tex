\begin{remark*}
The metric-space chapter now uses a Mathlib-first architecture. The old
hand-rolled metric structures are retained only as a learning model, under the
names \texttt{ScratchMetric} and \texttt{ScratchMetricSpace}.

The scratch model is bundled: a value of type
\[
  \texttt{ScratchMetricSpace Point}
\]
contains a chosen carrier type, a concrete distance function, and proof fields
for nonnegativity, identity of indiscernibles, symmetry, and the triangle
inequality. This is useful pedagogically because it exposes the raw proof
obligations behind the phrase ``is a metric.''

The main formalization, however, uses Mathlib's unbundled typeclass interface:
\[
  \texttt{variable \{X : Type u\} [MetricSpace X]}.
\]
Here \texttt{X} is the carrier type and \texttt{[MetricSpace X]} asks Lean to
find a metric-space instance through typeclass search. Once that instance is in
scope, the distance function is Mathlib's global notation
\[
  \texttt{dist x y}.
\]
The standard theorems are also available immediately, such as
\texttt{dist\_self}, \texttt{dist\_comm}, \texttt{dist\_nonneg}, and
\texttt{dist\_triangle}.
\end{remark*}

\begin{remark*}
The architectural distinction is:
\[
\begin{array}{c|c}
\text{Scratch model} & \text{Mathlib model}\\
\hline
\texttt{ScratchMetricSpace X} & \texttt{[MetricSpace X]}\\
\texttt{M.distance x y} & \texttt{dist x y}\\
\text{local bundled proof fields} & \text{global API lemmas}\\
\text{teaches obligations} & \text{supports theorem proving}
\end{array}
\]

Mathlib's \texttt{MetricSpace} is not merely a record with a distance function.
It extends \texttt{PseudoMetricSpace}, and through that hierarchy it carries
compatible uniform, bornological, and topological structure. Consequently, a
type with \texttt{[MetricSpace X]} can automatically be used as a topological
space. This is why Mathlib's design is powerful: metric facts, uniform-space
facts, and topological facts interoperate through the same typeclass instance.

The scratch-to-Mathlib bridge in Lean is therefore not just a matter of copying
a distance function. Mathlib also expects the topology to be compatible with
that distance. The helper
\[
  \texttt{ScratchMetricSpace.toMathlibMetricSpace}
\]
demonstrates this boundary: it converts a scratch metric space to a Mathlib
\texttt{MetricSpace} only after a compatible topology has been supplied.
\end{remark*}

\begin{remark*}
The active theorem files should use Mathlib notation. For example, a theorem
about arbitrary metric spaces should start from a context such as
\[
\begin{gathered}
  \texttt{variable \{X : Type u\} [MetricSpace X]}\\
  \texttt{variable (x y : X) (r : Real).}
\end{gathered}
\]
Then open balls and closed balls are not new structures. They are sets:
\[
  \texttt{Metric.ball x r : Set X},
  \qquad
  \texttt{Metric.closedBall x r : Set X}.
\]
This matches the way ordinary mathematics treats a ball: it is a subset of the
ambient metric space, not a separate object with its own fields.
\end{remark*}
